## 🏢 企业风控 · 保险场景借鉴

### 场景 1：把「Agent Skills」纳入企业 SBOM
[superpowers](https://github.com/obra/superpowers) / [agent-skills](https://github.com/addyosmani/agent-skills) 让每个 Agent 都是「一束 Skill 的组合」。**风控建议**：
- Skills 也要 SBOM——记录每条 Skill 的来源、版本、签名；
- 引入「Skill 风险评级」：能调外网 / 能写文件 / 能转账 / 能改人事档案，分级授权；
- 复用 [maigret](https://github.com/soxoj/maigret) 的"账号画像"思路，给企业内每个 Service Account（包括 Agent）建立"账号血缘 + 行为画像"，反欺诈基础。

### 场景 2：把 [Fable 30 天数据留存](https://support.claude.com/en/articles/15425996-data-retention-practices-for-mythos-class-models) 写进合同
- 重新审查与 Anthropic / OpenAI 的 DPA（数据处理协议）；
- 客户敏感字段（保单、医疗记录、被保人 PII）走「本地脱敏 → 远端调用」的两段式；
- 同时给业务方做培训：哪些 prompt 内容**绝对不能**进入需要留存的模型。

### 场景 3：参考 [LWN 案例](https://lwn.net/SubscriberLink/1077035/c7e7c14fbd60fae9/) 建「Agent 准入与熔断」
保险/银行内部 Agent 跑飞的代价更高：自动核保 / 自动理赔 / 自动催收 / 自动报税。**建议三件套**：
- 准入：每个 Agent 上线前必须有「最大行动半径」声明（最多影响 N 个保单、最多调用 N 次外部接口）；
- 熔断：超阈值自动转人工审批；
- 复盘：每周强制审计 Agent 决策与人工决策的偏差。

### 场景 4：[百万 Agent 互联](https://www.technologyreview.com/2026/06/11/1138794/google-deepmind-is-worried-about-what-happens-when-millions-of-agents-start-to-interact/) 与保险新险种
DeepMind 担心的是「Agent 群涌现风险」。对保险公司这是双面机会：
- **承保侧**：研发新险种——「Agent 责任险」「Agent 业务中断险」「Prompt 注入险」；
- **理赔侧**：建立「Agent 事故定责」模型——是模型方、还是 Skill 开发者、还是部署方？参考 [Anthropic Fable 护栏争议](https://techcrunch.com/2026/06/10/cybersecurity-researchers-arent-happy-about-the-guardrails-on-anthropics-fable/) 思考责任划分；
- **再保侧**：和云厂商共建 Agent 风险敞口共保。
