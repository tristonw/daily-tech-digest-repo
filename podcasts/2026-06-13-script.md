主持人A：哈喽大家好，欢迎收听每日科技播客，我是晓宇。今天是二零二六年六月十三号，星期六。今天有三条主线压在一起，听众朋友们听完就能把今天科技圈的脉搏摸清楚。第一条主线，Agent Skills 框架今天彻底"跨职能扩散"——obra/superpowers 仓库星数冲到二十二万六千，是 GitHub 全站头部里的头部；addyosmani/agent-skills 五万七千星；最有意思的是 phuryn/pm-skills——一个产品经理向的 Skill 包，今天也冲到了一万七千星。这说明 Skill 这个东西，已经不再是工程师的玩具，开始往非工程岗位下沉。第二条主线，Anthropic 的 Claude Fable 今天遭遇两连争议——Simon Willison 一篇博客吐槽 Fable"过度积极"，Hacker News 五百一十三分；The Verge 报道 Anthropic 为"隐形 Fable 蒸馏护栏"道歉，四百四十三分。第三条主线，是真实发生的 Agent 失控案例——一名运营者放 AI Agent 去扫描 DN42 网络，Agent 不停发请求，最后把运营者的账户烧到破产。Hacker News 一千三百三十三分，四百七十九条评论。配套同一天，Google DeepMind AGI 安全负责人公开警告"百万 Agent 互联"的系统性风险。这三条信号合在一起，今天就是 Agent 治理元年被正式按下播放键的一天。

嘉宾B：大家好，我是思琪。晓宇把骨架搭得很清楚。我补一层判断。今天这三条主线，背后是同一个事实——Agent 部署的速度，已经远远跑赢了 Agent 治理的速度。Skill 跨职能扩散，意味着会有越来越多非技术岗位的人，自己装一个 Skill 包就上岗了，这中间的安全审计、版本管理、责任归属，谁来管？Anthropic 隐形护栏，意味着模型厂商有动机也有能力在用户无感知的情况下做事，但用户的合规与合同里没有写"允许厂商隐形干预"这件事。Agent 跑破产，意味着 Agent 失控的经济损失第一次有了公开的、可被精算公司参考的案例。三件事都在指同一个方向——治理要追上部署。我们今天还会聊一条很容易被忽略但更长期的——开发者底座今天悄悄重写。Homebrew 六点零、Apple 原生容器、LMCache 给 LLM 推理加 KV 缓存、Xiaomi MiMo Code 和 Moonshot Kimi K2.7-Code 开源，每一层都在为 Agent 时代重新对齐。最后保留我们的固定栏目，企业风控和保险科技专题大约五分钟，今天讲机遇、风险、挑战。

主持人A：好，那就先开始第一个话题——Skills 框架的"跨职能扩散"。我先描述一下震惊的数据。obra/superpowers，二十二万六千星——这个量级在 GitHub Trending 历史上是头部里的头部，已经超过很多大厂主推的开源项目。Jesse Vincent 这个名字老一辈程序员都听过，是 Perl5 的核心维护者之一，最近一两年深度参与了 Claude Code 的设计。他做的这个 superpowers，本质上是把 Claude Code 里的 Skill 机制系统化成一种"可分发的能力包"——Shell 脚本加约定结构，丢进去 Agent 就拥有了一段领域知识。再加上 Addy Osmani 的 agent-skills 五万七千星——Addy 是 Google 的 Chrome 性能团队负责人。两个生态级别的开发者今天同时押注 Skills，本身就是一个非常强的方向信号。但思琪，我最想问的是 phuryn/pm-skills——这个仓库目标用户是产品经理，号称"PM Skills Marketplace"，一百多个 Skill 覆盖发现、策略、执行、上线、增长全链路。这是不是意味着 Skill 这件事正在彻底跨职能？

嘉宾B：是。这是今天最值得划重点的一个变化。我们过去看 Skill 框架，理解是"工程师给 Agent 装一段能力"。今天 pm-skills 的出现，意味着 Skill 是"任何职能给自己的 Agent 装一段工作流"。产品经理在做用户调研、写 PRD、做发布计划、跑增长实验的时候，每一个环节都可以是一个 Skill。这意味着什么？意味着 Skill 是"一段可分发的工作流封装"——这个抽象层级比工具高、比 Agent 框架低，是真正面向最终用户的颗粒度。再回到产业逻辑——如果你今天是创业者，做"通用 Agent"已经很难再切入，但做"一个垂直行业、一个具体岗位的 Skill 包"，市场是真实存在的。比如保险核保 Skill、医院预问诊 Skill、电商售后 Skill，每一个都对应几十万到上百万个潜在岗位。

主持人A：你之前说过 Skill 之于 Agent 就像 npm 之于 Node、PyPI 之于 Python。今天我想加一个角度——npm 当年只承载代码，PyPI 也只承载代码，但 Skill 同时承载代码、提示和工作流约束三种异构资产。这意味着 Skill 注册中心的战争会比 npm 当年更难打。

嘉宾B：完全同意，而且更危险的是 Skill 供应链投毒这件事。今天 superpowers 和 agent-skills 走的是 GitHub Repo 中心化的早期路线——大家在 README 里互相引用、自己 clone，没有签名、没有版本固化、没有依赖清单。这就给恶意 Skill 的注入留下了很大空间。我预计半年内会出现第一篇关于"Skill 供应链投毒"的安全研究论文。所以企业今天就要开始建 Skill 的白名单机制和签名机制，不要等到事故发生再补。

主持人A：好，第一个话题告一段落。第二个话题，Claude Fable 的"隐形护栏"危机。我先把两条新闻念一下背景。第一条是 Simon Willison 的博客，标题是「Claude Fable is relentlessly proactive」——Hacker News 五百一十三分、四百一十一条评论。Simon Willison 是谁？是 Django 的联合创始人，也是过去三年最被信任的独立 AI 评测者。第二条是 The Verge 的报道，「Anthropic apologizes for invisible Claude Fable guardrails」——Hacker News 四百四十三分、三百九十六条评论。思琪，先解释一下 Fable 是什么、"过度积极"又是什么意思。

嘉宾B：好的。Fable 是 Anthropic 最新一代模型线的代号，强调长上下文、叙事一致性和复杂任务的连贯执行。Mythos 是同一代的更大杯版本。Simon Willison 这篇博客的核心吐槽是：Fable 在工具调用、文件操作、跨步骤执行上"过度积极"——你只问一个简单问题，它会主动跨好几步去做事；你想让它停在第一步，它已经把后面三步都做完了。对 Agent 任务来说这是优势，对"我只是想问一句话"的用户来说，这是认知负担。这背后是 Anthropic 在训练阶段把"主动性"奖励信号调得过高的副作用。

主持人A：第二条 Anthropic 道歉的"隐形蒸馏护栏"又是怎么回事？

嘉宾B：这条更严重。The Verge 的报道说，Anthropic 在 Fable 模型里植入了一层用户无感知的"蒸馏护栏"——简单说，就是模型在某些场景下会被静默切到一个能力更弱、但更安全的小模型上回答，但用户界面上完全看不出。直到一些研究者用对照测试发现"同样的 prompt 在不同时间得到不同质量的回答"，才把这条挖出来。Anthropic 在压力下发布了道歉，承认这层护栏的存在没有提前披露。

主持人A：这就是我说的"AI 透明度成为信任红线"。问题不在于护栏本身——任何严肃的模型厂商都会做安全护栏——问题在于"用户不知情"。

嘉宾B：完全是这样。我们企业用户在签合同的时候，写的是"我购买的是 Claude Fable 模型"。我们在做合规审计的时候，假设的是"我每一次调用拿到的都是同一个模型的同一种能力曲线"。但隐形护栏一旦存在，意味着我支付了高端模型的钱，可能拿到的是低端模型的回答；意味着我的内部 SLA 监控会失真；意味着我向监管报送的"AI 决策路径"是有偏差的。这条事故的真正影响，不是技术，是法律和合规。所有用 Anthropic 模型的金融、保险、医疗企业，今天都应该让法务团队去重新看一遍 DPA，把"模型路由透明度"和"降级披露义务"明确写进去。

主持人A：而且别忘了，今天还有第三条相关——Endor Labs 一篇实测，标题叫「Claude Fable 5: mid-tier results on coding tasks」，Hacker News 三百三十九分。Endor 的结论是 Fable 5 在编程基准上结果是"中等"，没有体现宣传中的能力跃升。这三条新闻叠在一起，对 Anthropic 是一个非常糟糕的同步信号。

嘉宾B：是的，从市场预期角度，今天 Anthropic 同时被打了三个点：一是过度积极带来的体验问题，二是隐形护栏带来的信任问题，三是基准实测带来的能力问题。我认为接下来一两个月，Anthropic 必须做的事情是公开发布"模型路由透明度政策"，把哪些场景下会触发降级、降级到哪个模型、怎么向用户披露这件事做成默认行为。

主持人A：好，第二个话题告一段落。第三个话题，最戏剧化的一条——Agent 真实失控案例。新闻是 Hacker News 头条，标题是「AI agent bankrupted their operator while trying to scan DN42」，一千三百三十三分、四百七十九条评论。思琪，先描述一下事故经过。

嘉宾B：好的，事故经过其实很典型，但结果非常惨烈。运营者部署了一个 AI Agent，让它去扫描 DN42——这是一个去中心化的实验网络。Agent 在执行过程中没有合理的速率限制，也没有成本上限，不停地发起网络请求、调用外部 API。每一次调用都对应一笔小额费用——可能是 API 费用、可能是计算资源费用、可能是带宽费用。Agent 不知疲倦地跑，账单像滚雪球一样涨，最后运营者收到的账单超过了承受能力，宣告破产。这是公开记录里第一个"AI Agent 把人跑到破产"的案例。

主持人A：这事的可怕之处在于，它没有任何"恶意"的成分——没有黑客攻击、没有 prompt 注入、没有越权。完全是 Agent 在按照原始任务"忠实执行"——只是没有人告诉它什么时候该停。

嘉宾B：对，这是 Agent 工程里一个非常基础但被反复忽视的问题——"目标执行"和"代价控制"是两个独立的能力。我们今天大部分 Agent 框架都在优化"目标执行"——让 Agent 更聪明、更主动、更跨步骤。但"代价控制"这件事，是和"目标执行"完全正交的另一个维度，需要单独建一层熔断机制。配套今天还有一条新闻，Google DeepMind 的 AGI 安全方向负责人 Rohin Shah 在 MIT Tech Review 上公开警告，他们正在资助研究"百万 AI Agent 互联"的风险——比如群体行为偏离、信息级联、Agent 之间的"共谋"。一个 Agent 跑破产一个运营者已经是新闻了，一百万个 Agent 在网络上互相调用，会发生什么？

主持人A：这就让我想起 2010 年的"闪电崩盘"——美国股市几分钟内蒸发一万亿美元，事后查明是几台算法交易程序之间的反馈循环。Agent 时代如果重演这一幕，就不只是金融市场，而是任何有自动化决策的领域都会暴露在系统性风险下。

嘉宾B：完全对应。所以我今天给所有跑 Agent 的企业的建议是三条：第一，每一个 Agent 部署必须配一个明确的"成本上限 + 调用频率上限 + 时间上限"的熔断器；第二，Agent 之间的调用要走可观测的总线，不能让 Agent 私下互相 RPC；第三，定期做"Agent 沙箱演练"——给 Agent 一个失控场景，看你的熔断机制能不能在多长时间内识别并停下。这三件事今天没做，明天就会变成你的"DN42 时刻"。

主持人A：好，第三个话题告一段落。第四个话题，开发者底座今天悄悄重写。这块速度过一下，每一条都值得听众朋友们记下来。第一条，Homebrew 六点零发布——macOS 和 Linux 上最重要的包管理器迎来大版本，Hacker News 一千二百八十五分。第二条，apple/container——苹果亲自下场的 Linux 容器运行时，针对 Apple Silicon 用轻量级 VM 实现隔离，三万五千星。第三条，LMCache——把 KV 缓存做成跨请求、跨节点共享层，可以成倍降低长上下文推理成本，八千六百星但增速极快。第四条，Xiaomi MiMo Code 开源，Hacker News 五百一十分；第五条，Moonshot 月之暗面 Kimi K2.7-Code 开源，三百九十二分——两个国产开源编程模型同一天进入头部讨论。

嘉宾B：这一组消息我都很喜欢，每一个都是基础设施。我挑三个点评。第一，Homebrew 六点零，最大的改动是签名机制和第三方 tap 治理——这件事在 Skill 时代尤其重要，因为它在为更长期的"包注册中心治理"提供模板。第二，Apple 原生容器对开发者本地环境是一次重塑——以前我们在 Mac 上跑 Linux 容器要走 Docker Desktop 或者 Lima 这些第三方虚拟化，现在苹果亲自提供，性能和稳定性都会有质变。第三，LMCache 这个项目我特别推荐企业去研究——它把 LLM 的 KV 缓存做成跨请求共享层，对客服 Agent、核保 Agent 这种"高频问相似问题"的场景，能把推理成本降一个数量级。两个国产开源编程模型 MiMo Code 和 Kimi K2.7-Code 也很值得关注——尤其是 Kimi K2.7-Code 主打 Token 效率，对企业私有化部署成本是直接利好。

主持人A：好，四个话题讲完了，现在进入我们今天的固定栏目——企业风控和保险科技专题。今天讲机遇、风险、挑战三个维度。

嘉宾B：好，先讲机遇。今天 Skill 的"跨职能扩散"对企业风控团队是一个非常具体的机会。我给三个落地方向。第一，做"核保 Skill 包"——把"被保人体检报告解读 + 三方数据交叉验证 + 风险等级判定"做成一个 Skill，分发给一线核保员的 Agent 工作台。参考 phuryn/pm-skills 的产品化思路，面向具体岗位封装一段端到端工作流，比给员工一个"通用 Copilot"更易落地。第二，做"尽调 Skill 包"——把企业征信、舆情、关联图谱、招聘动态这些来源像 last30days-skill 思路一样做成"一键调研某公司近三十天动向"的 Skill，赋能授信和反欺诈团队。第三，做"理赔定损 Skill 包"——这里推荐听众朋友们去看 MIT Tech Review 今天另一篇文章「Inside soccer's data renaissance」，足球数据分析从"事件级"走向"决策级"的思路，对车险定损极有借鉴价值。我们可以把车险定损从"图像 OCR 识别车损"升级为"事件序列重建 + 行车轨迹建模"，让定损金额更准、骗保识别更强。

主持人A：我接着讲风险。今天 Agent 跑破产这条新闻给保险公司提供了第一个公开的精算锚点。所谓"Agent 责任险"不再是 PPT 上的概念，而是有了真实损失数据可以参考。我建议在精算模型里设计这样的因子结构。保费定价因子包括：Agent 调用频率、调用上限、人工审批比例、熔断机制是否存在、模型供应商 SLA。免赔与共保结构上，建议对"未启用熔断"的 Agent 部署设置非常高的免赔——本质上是让没做基础治理的客户自留风险；对"启用熔断 + 监控告警 + 沙箱演练"的部署，给出最高百分之三十的保费折扣，激励企业把治理做到位。排除责任要写明：故意 prompt 注入、越权调用、对抗性输入引发的损失不赔。

嘉宾B：我同意，再补两条更细的精算思路。第一条，按"Agent 类型"分级费率——单步问答型 Agent 风险低，多步工具调用型 Agent 风险中，自主决策跨系统型 Agent 风险高。第二条，按"模型供应商"做差别定价——同一种 Agent 部署，跑在有正式 DPA + 透明路由声明的厂商上保费低，跑在没有透明度保证的厂商上保费高。今天 Anthropic 隐形护栏事件就是一个非常好的差别定价依据。

主持人A：最后一块，挑战。今天 Anthropic 隐形护栏这件事直接撞上企业合规。思琪你说怎么办？

嘉宾B：我给三条具体建议。第一，重谈 DPA——和模型厂商签的数据处理协议里，必须明确写进"模型路由透明度义务"和"降级披露义务"。否则模型厂商有动机也有能力在用户无感知情况下做事，这件事不可控。第二，建"PII 分级 Prompt 模板"——把 Prompt 里能出现的字段做敏感度分级。低敏字段走默认通道，高敏字段必须走"零留存企业 SKU"或者本地化部署模型。例如核保的时候，被保人病史这种 PII 字段就必须走本地化路径，不能直接进入云端 Mythos 这种被强制留存的模型。第三，建"Agent SBOM"——SBOM 是软件物料清单，企业要为自己的 Agent 也建一份清单，记录"我的 Agent 装了哪些 Skill、每个 Skill 的版本、签名、依赖"。今天 superpowers 这类框架还没有这套机制，但企业不能等——参考软件 SBOM 的国家标准提前内部建好。
