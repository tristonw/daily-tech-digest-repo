主持人A：哈喽大家好，欢迎收听每日科技播客，我是晓宇。今天是二零二六年五月三十一号，星期日。咱们照例用十几分钟，把这两天科技圈最值得记的事情捋一遍。
嘉宾B：大家好，我是思琪。今天的信息密度依旧很高，我先给个地图。我自己看下来，今天有五条线值得抓——一条是协议：MCP 这个名字今天在 Hacker News 上被人贴上了"死了？"的标签；一条是工具：GitHub 热榜继续被"技能化"的项目占领；一条是架构：一篇用 SQLite 做持久化工作流的文章冲到了 HN 第二；一条是资本：OpenRouter 拿到一点一三亿美元的 B 轮；最后一条是组织：MIT Technology Review 给所有想 Agent 化的企业泼了一盆冷水。
主持人A：协议、工具、架构、资本、组织，五条线，听起来今天确实是把 Agent 这一栈从底到顶又过了一遍。我们一条一条来。先说最有冲突感的——协议层。
嘉宾B：对。Hacker News 今天有一篇引爆讨论的文章，标题就叫《MCP is dead?》，热度三百六十一，评论三百四十四。
主持人A：去年这个时候 MCP 还在被各大厂当作 Agent 接入层的统一协议在推，怎么一年不到就有人贴"死了"的标签了？
嘉宾B：文章本身不是来唱衰的，是来复盘的。作者团队从去年开始把 MCP 当主线接入方式做了一年多，最后得出的结论很务实——在大量的生产场景里，MCP 多了一层抽象，但带来的可调试性、可控性反而下降了。他们最后选择回到"显式 API + 模型自己写代码"的方式，速度快、上下文清楚、出问题也好查。
主持人A：所以本质上是协议层在被工具层挤压。
嘉宾B：你这句话说到关键。我们顺势看一下今天 GitHub 热榜——你猜猜前十的关键词是什么？
主持人A：技能？Skill？
嘉宾B：宾果。我念几个：第二名是 affaan 斜杠 ECC，热度十九点九万，副标题是"Agent harness 性能优化系统"，主打把 Skills、本能、记忆、安全、研究优先开发这些模式打包，跨 Claude Code、Codex、Opencode、Cursor 通用；第四名是 anthropics 斜杠 skills，十四点四万，Anthropic 官方的 Agent Skills 仓库；第十三名是 taste-skill，二万八千多颗星，名字特别有意思——它的卖点就一句话，"不要让你的 AI 生成那种千篇一律、套路化的内容"。
主持人A：Anthropic 官方、第三方框架、风格控制——分层挺明显的。
嘉宾B：对，这就是"生态在长出来"的样子。底层是 ECC 这种通用的 Agent harness 框架；中间是 anthropics skills 这种官方标准库；上面是 taste-skill 这种特定用途的能力封装。再往上还会有"行业垂直 Skill"——保险反欺诈 Skill、医疗摘要 Skill、合同审阅 Skill，这是未来一年最值得抢的位置。
主持人A：那回到 MCP 那篇文章——你怎么解读"协议"和"技能"这两条线的关系？
嘉宾B：我的解读是——协议之争未必能用一份"通用接口"一战定胜。更可能演化成"每个领域一套可分发的 Skill 包 + 显式 API 调用"这种实用主义的形态。Skill 是组装单元，API 是调用方式，协议反而退到了背后。
主持人A：好，第一条线收尾——结论是，**Agent 接入层正在从"协议"转向"技能 + 显式 API"**，做企业内部 Agent 平台的同学要早点把这件事考虑进架构里。我们看第二条——架构层。你说有篇 SQLite 的文章冲到 HN 第二？
嘉宾B：是的，标题就叫《SQLite is all you need for durable workflows》，"持久化工作流，用 SQLite 就够了"。热度六百四十八，评论三百四十六。
主持人A：这个标题我想起几年前那个 "Just Use Postgres" 的老梗。是不是又一次反复杂化运动？
嘉宾B：方向是一样的，但比 Postgres 那波还要更极致。作者主张：很多团队在做长任务、做 Agent 编排时，第一反应是上 Temporal、上 Airflow、上 Cadence，但其实在中小规模下，一个 SQLite 加上幂等键、状态机、定时唤醒，就能解决百分之八九十的需求。
主持人A：这对那些没有专职平台团队的公司，是个特别及时的提醒。
嘉宾B：对。你想，今天 Agent 这一波最大的隐患就是"中间件爆炸"——一个长流程，要拉向量数据库、要拉工作流引擎、要拉消息队列、要拉对象存储、要拉评估平台。每多一个组件，运维成本、故障面、合规面都翻一倍。
主持人A：所以"统一栈"反而是种克制。
嘉宾B：而且和今天另外两个项目能对得上。第一个是 microsoft 斜杠 markitdown，已经十三万星，把 Office、PDF、各种文档转成 Markdown；第二个是 OpenBMB 斜杠 VoxCPM，是一个无需 Tokenizer 的多语种 TTS。前者是前置管道，把外面世界变成 LLM 能吃的格式；后者是后置出口，把 LLM 的输出变成本地化的语音。再加上 SQLite/Postgres 做状态、Skill 做能力——整个 Agent 栈在收敛。
主持人A：少了很多以前以为必须的组件。这个收敛对企业意义很大——意味着上 Agent 的门槛和试错成本都在快速下降。我们看第三条——资本层。
嘉宾B：今天最值得一聊的是 OpenRouter 的 B 轮。一点一三亿美元，公告今天上了 HN，热度三百二十七，评论一百四十九。
主持人A：OpenRouter 我们之前提过——它是个模型路由层，对外一份合同、一个账户，对内可以接 OpenAI、Anthropic、Mistral、Google 等几十家模型。
嘉宾B：对。我觉得这一轮的意义不在金额本身，而在"它确认了一个新品类"。在过去两年，AI 基础设施的赛道分化得越来越细——向量数据库、Agent 评测、推理加速、合成数据，现在多了一个"模型路由 / 多供应商接入层"。每一个赛道都被资本一笔一笔地盖章。
主持人A：对企业意味着什么？
嘉宾B：意味着"一份合同接 N 家模型"会变成主流采购形态。CIO 不再需要去和每家模型公司单独谈商务、单独做发票、单独做配额管理，而是通过路由层一次拿到。这对企业内部的预算管理、成本控制、和合规审计都是利好。
主持人A：那也带来风险——把所有模型供应商都集中到一个路由层，反而把单点风险又抬高了一档。
嘉宾B：你说到点子上了。我们等会儿讲风控专题的时候，会专门讲怎么治理这件事——简单说就是"对外只签一个路由层，对内仍然要支持多路由层并行兜底"。先埋个伏笔。
主持人A：好，第三条线收尾。我们看第四条——组织层。
嘉宾B：MIT Technology Review 这周连续出了几篇，最关键的是这一篇——《Rethinking organizational design in the age of agentic AI》。这篇引用了一份调查，给了一组特别扎眼的数字：**百分之八十五的企业说他们想在三年内实现 Agent 化；同时百分之七十六的企业承认，自己当前的运营和基础设施根本撑不起这件事**。
主持人A：这两个数字放在一起，杀伤力太大了。
嘉宾B：而且 MIT 这周还有一篇姐妹文章——《puncturing the AI jobs panic》，主张到目前为止 AI 取代白领的大规模实证证据仍然稀薄。两条放在一起读，结论很清楚——**焦虑被高估，落地被低估**。
主持人A：那瓶颈到底在哪儿？
嘉宾B：不在模型，在人、流程、和工作流。模型层每隔两个月就跳一档，但组织里的流程文档、权限、SOP、可观测性，可能还停留在十年前。你把一个最先进的 Agent 接进一个"流程都说不清楚"的组织，它只会把混乱放大。
主持人A：所以下一步该做什么？
嘉宾B：先把流程数字化、可观测化、权限可控化。先把"哪一步谁做、用什么数据、产出什么"梳清楚，再上 Agent。否则就是花更多钱买更快的速度，去把错误执行得更彻底。
主持人A：好。最后一条——产品形态层，今天有两个项目我也想聊一下。
嘉宾B：可以。一个是 ruvnet 斜杠 RuView，热度六万八千多，副标题特别有意思——"把商品 Wi-Fi 信号变成实时空间智能、生命体征监测、存在检测，一像素视频都不需要"。另一个我们前面提过的 OpenBMB 斜杠 VoxCPM，多语种、无 Tokenizer、可做创意配音和真人级克隆。
主持人A：这两个项目放在一起讲什么？
嘉宾B：讲的是"另一条产品线在长出来"——AI 不一定都长得像 ChatGPT 那样有一个对话框。RuView 把整个房间变成 sensor，没有摄像头、没有画面，但你能知道有没有人在、谁摔倒了、谁的呼吸频率不对。这种被动感知 + 隐私友好的方案，对养老监护、商业地产、智能办公保险，全是直接的应用场景。
主持人A：VoxCPM 这边呢？
嘉宾B：VoxCPM 让"本地化语音"变成可能——客服质检不用上云，培训配音不用上云，本地播客也不用上云。对受合规约束的金融和保险行业特别友好，是国内合规场景里少见的"开箱即用 + 真人级"组合。
主持人A：好，前面五条线我们过完了——协议、工具、架构、资本、组织、产品形态。下面我们进入今天的核心环节：**企业风控与保险科技专题**。
嘉宾B：好，这部分我想给在做核保、定损、反欺诈、合规、内审的朋友列八条非常具体的、今天就能开始想的事。
主持人A：八条挺多，咱们快节奏过。
嘉宾B：**第一条，把内部合规和业务规则做成"Skill 包"**。借鉴今天热榜的 anthropics 斜杠 skills 和 ECC。反洗钱筛查、KYC 文档抽取、被保险人病历摘要、车险事故图像分类——每一项都封成一个可版本化、可审计、可回滚的 Skill，而不是散落在几十个 Prompt 文件里。
主持人A：这样治理就跟着上来了——每个 Skill 谁审过、用了哪一版、出了问题回滚到哪一版，都有迹可循。每一条决策也能定位到"是哪一版 Skill 在哪条数据上做出的"。
嘉宾B：对。**第二条，把核保和理赔这种长流程，先用 SQLite 或 Postgres 跑通，别一上来就上 Temporal**。借鉴今天 HN 第二那篇文章的精神。核保从受理到出单可能持续数小时甚至数天，跨多个外部数据源——单实例数据库就能跑、状态可重放、审计就在同一张表里，省去重型工作流引擎的运维成本。
主持人A：而且要换的时候再换，沉没成本小。第三条。
嘉宾B：**第三条，文档管道本地化**。markitdown 把单证、保单、批单、医院报告、车检报告转成 Markdown，VoxCPM 把通话录音/质检稿件转成本地化语音输出。两套都可以完全在内网跑，对数据出境合规要求高的金融机构特别友好。
主持人A：这个对很多需要把数据留在本地的金融机构，是即插即用的。第四条。
嘉宾B：**第四条，把"路由层"作为模型采购的标配，但别把单点风险也一起买了**。今天 OpenRouter 的一点一三亿美元 B 轮提醒大家——"对外签一个路由层供应商"是趋势。但对内，建议保留至少两条独立通路：一条走商业路由层（比如 OpenRouter），一条走自建网关 + 直连主要模型厂商。一旦路由层出故障、价格策略变化、或者出于合规原因要切供应商，企业有 24 小时之内切换的能力。
主持人A：第五条。
嘉宾B：**第五条，把"住宅代理风险评分"作为反欺诈/反爬定价的独立维度**。今天 Ars Technica 报了一条很重要的新闻——一个超过一千七百万设备的住宅代理僵尸网络被联合执法拆除，背后据报道与俄罗斯的住宅代理网络有关。
主持人A：这一条对保险和银行的反欺诈意味着什么？
嘉宾B：意味着你过去十二个月接到的"看起来正常的家庭 IP"，可能很大一部分其实是被劫持的设备。做反欺诈服务定价、做网络责任险定价的时候，"住宅 IP 是不是干净"应该是一个独立分数，而不是和"匿名代理"混在一个维度里。
主持人A：第六条。
嘉宾B：**第六条，把 AI 内容溯源信号接进风控决策树**。今天 Ars Technica 还有一篇——一家创业公司用"免费保洁换录像"的方式给机器人采训练数据。家用录像、家财险报损照片、车险事故视频、企业宣传素材——这些都将带上"是否 AI 生成、是否被篡改、是否被授权再用"的元数据。我们要做的事——把这类信号接进反欺诈决策树，并和保单条款里"训练数据可撤回、可二次使用边界"提前对齐。
主持人A：第七条。
嘉宾B：**第七条，组织准备度先行**。回到 MIT 那 85% 与 76% 的数字。**别先买 Agent 平台，先把流程、权限、可观测做扎实**。一个非常具体的建议：在公司内部找两到三个"已经有清晰 SOP、有完整日志"的小流程——比如团险出单中的某一个环节、车险定损中的某一类小额案件——先用 Agent 跑通这两个，再扩。比"一次性大改造"靠谱十倍。
主持人A：最后一条。
嘉宾B：**第八条，航天和巨灾保险定价的信号源升级**。今天 Ars Technica 那篇《Rocket Report: A dark day for Blue Origin》和姐妹篇《Why the failure of Blue Origin's New Glenn rocket is so catastrophic》——讲的是 Blue Origin 的 New Glenn 出了一次严重的失利。**单一发射保单、单一供应链事件**对再保险池的冲击在变大。建议把开源情报、行业新闻、社媒舆情都纳入累积控制的信号源，并用 Agent 做定期复盘——而不是等到季度评审才发现风险敞口已经超标。
主持人A：八条总结下来，我感觉今天的风控专题其实有一个共同主线：**把每一个能产生数据的环节都数字化、把每一个能产生风险的环节都做成可审计的产品、对外只签一个供应商但对内永远留两条路**。
嘉宾B：你这个总结非常到位。一句话——不要追"全自动"，要追"可治理"。
主持人A：好，时间差不多了。我们用一个习惯的环节收尾——今日金句。
嘉宾B：我用今天的内容拼一个——"协议在被工具替代，框架在被技能替代，重型工作流在被一张数据库替代。Agent 时代真正的护城河，不是更复杂的栈，而是更克制的栈和更清晰的流程。"
主持人A：很好，加一句我的版本——"百分之八十五的企业想 Agent 化，百分之七十六承认自己没准备好。准备度，仍然是 Agent 时代最被低估的资产。"
嘉宾B：完美收尾。
主持人A：今天就到这里。我们明天同一时间再见，拜拜。
嘉宾B：拜拜。
