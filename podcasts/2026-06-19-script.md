主持人A：哈喽大家好，欢迎收听每日科技播客，我是晓宇。今天是二零二六年六月十九号，星期五。先给大家一个直接的开场——昨天我们讲的核心命题是"AI 标签的红利在退潮，基础工具栈出现系统性松动"。今天这两条线都在继续推进，但今天最值得我们重点拆解的，是另外一个更早期的信号：通用大模型公司开始**集体押注垂直化第一站**。Midjourney 正式发布了 Midjourney Medical，专门做医疗影像，今天在 Hacker News 上一千二百四十四分、八百三十一条评论，是今日全场最热。同一天，DeepSeek 也推出了 Vision 多模态版本，四百二十三分、一百六十七条评论。这两件事单独看是新闻，合在一起就是趋势——**全球最有钱、最擅长打价格战的两家通用大模型公司，同一天把"垂直化第一站"押在了视觉与医疗**。思琪，这件事咱们今天先重头拆。

嘉宾B：哈喽大家好，我是思琪。今天确实要先聊 Midjourney Medical 与 DeepSeek Vision。这两条新闻同一天出现，绝对不是巧合。我们先把背景拉一下——二零二三年到二零二五年，通用大模型公司打的是"模型参数 + 训练数据 + 服务速度"的三方面竞赛，毛利率非常高。但是到二零二六年，**通用 LLM 已经商品化了**——同样能力的模型，价格降到原来的十分之一，毛利率被压扁。这种情况下，所有玩家的下一步必选项就是垂直化。问题是——垂直化的第一站为什么是医疗加影像？

主持人A：我觉得这里有三个原因可以讲透。**第一个原因——数据壁垒高**。通用 LLM 谁都能训练，但医疗影像数据需要合作医院、合作放射科、合作政府监管机构，谁先建立这套关系网，谁就占据了三到五年的护城河。**第二个原因——监管壁垒就是护城河**。FDA 在美国、NMPA 在中国、EMA 在欧盟，每一个监管机构对医疗 AI 设备的审批流程都是长周期的，这恰好让先发玩家拿到时间窗口。**第三个原因——技术距离短**。从"通用图像生成"到"医疗影像理解"，技术栈高度相邻，过去两年 Midjourney 在通用图像生成上积累的模型架构、训练流水线、图像理解能力，可以直接迁移。

嘉宾B：对，这三个原因合起来就解释了为什么医疗影像是垂直化的第一站。但更值得讨论的是——**这件事对保险行业的直接含义**。我个人判断未来 24 个月，健康险与重大疾病险的核保数据流会经历一次结构性重构。原本的链条是这样的——病人去医院 → 医院 PACS 系统拍片 → 影像中心放射科医生人工读片 → 出 PDF 报告 → 病人或医院把报告寄给保险公司 → 保险公司核保员人工读 → 核保结论。这个链条又长又慢又不结构化。

主持人A：未来的链条会变成什么样？我大胆猜一下——病人在医院拍片 → 影像数据直接上传到"医疗 AI 影像 SaaS"（可能是 Midjourney Medical、可能是 DeepSeek Vision、也可能是本土的本地化产品）→ 自动出结构化报告 → 通过 API 直接对接到保险公司的核保系统 → 核保结论几分钟出来。**这是健康险数字化最大的一次结构性变化**。

嘉宾B：所以对保险公司的直接建议是——Q3 必须启动技术准备。具体三件事——**第一件——选一到两家合作医院做 PoC**。先在重大疾病险的核保流程里试点一个细分场景，比如肺结节筛查、乳腺癌筛查、心血管影像评估。**第二件——建立"AI 影像报告"的核保认可机制**。哪些险种可用、可信度阈值如何设定、人工复核的覆盖比例是多少。**第三件——和供应商做技术能力评估**。Midjourney Medical、DeepSeek Vision、本地多模态模型厂商，三条路径都要评估——是 SaaS 模式好还是本地化部署好、合规边界在哪里。**早动手的险企能在 2027 年的健康险定价与核保效率上拿到 12 到 18 个月的领先窗口**。

主持人A：好，第一条话题就停在这里。我们切到第二条——今天 Ars Technica 报道了一个企业 IT 圈的重磅消息：**英国零售巨头 Tesco 把 40000 个服务器负载迁出 VMware**，并且在英国公开文件里直接披露 Broadcom 涨价 175%。这件事是昨天 HPE 用一年免费许可证挖墙脚故事的完美续集。

嘉宾B：我们先把数字放在背景里。Tesco 是欧洲最大的零售商之一，年营收超过 600 亿英镑，IT 规模在欧洲零售业里是顶级。40000 个服务器负载是什么概念——这不是一两个数据中心，是 Tesco 几乎全部的核心业务系统。一家零售巨头愿意公开披露涨价细节、迁出系统——**这是 VMware 替代故事的"主流化拐点"**。从昨天的 HPE 免费许可证到今天的 Tesco 公开文件，两件事拼起来证明——Broadcom 收购后的涨价策略已经触及大型企业的承受底线。

主持人A：我觉得这件事对保险行业的提醒非常强。保险公司风控、精算、客服三大类数据系统的虚拟化底座，过去十年大概率都是 VMware。**Broadcom 涨价的真实账单，2026 年下半年到 2027 年会陆续到**。Tesco 不是孤例，它只是第一个公开把这件事曝光出来的大型企业。后面会有更多的大型企业、保险公司、银行、电信运营商跟进。

嘉宾B：所以对保险公司 IT 部门的建议非常直接——**Q3 必须启动 VMware 替代 PoC**。三条最现实的路径——**HPE VM Essentials**（昨天讲过，免费一年许可证、Broadcom 直接对手）；**Proxmox**（开源、欧洲企业用得多、社区成熟）；**KVM 加红帽 OpenStack**（最经典的开源虚拟化栈、适合大规模数据中心）。三条都需要评估。**错过 2026 年 Q3 到 Q4 的窗口期，2027 年涨价账单到了再启动迁移，成本会指数级上升**。这件事的紧迫程度，CIO 应该直接列为本季度 Top 3 KPI。

主持人A：好，第二条话题就停在这里。我们切到第三条——昨天我们讲过 Epic Games 公开了自研版本控制系统 Lore，今天 Lore 不仅没掉下榜，反而 HN 分数从昨天的 877 涨到今天的 1166，连续两天霸榜。这件事本身就说明开发者社区对"Git 二十年统治终结"这件事的渴望被点燃了。

嘉宾B：今天的讨论里有两个新的焦点——**第一个焦点——会不会出现 GitHub-like 的 Lore 托管服务**。如果出现，那将是一个新的开发者基础设施市场。**第二个焦点——Lore 是否会原生支持 AI 模型权重的版本管理**。这是社区呼声最大的功能请求。如果 Lore 能正面解决"模型权重 + 训练数据 + 代码"统一版本管理的问题，**它有可能在 18 个月内取代 DVC、MLflow 这些凑合方案**。

主持人A：我想补充一条今天另外发生的相关大事——**RFC 10008 正式发布**，给 HTTP 协议新增了一个 QUERY 方法。这条新闻 384 分、160 条评论。你可能觉得"这是底层协议变化、跟我有什么关系"——但这件事的意义远大于表面。

嘉宾B：我来解释一下。过去十年所有 RESTful API 设计的工程师都遇到过这个问题——查询接口要带很多参数（比如复杂的搜索查询、报表参数），GET 方法 URL 太长会被截断、POST 方法又不是幂等的（缓存、重试都会出问题）。这个死胡同绕了十年。**今天 RFC 10008 终于给出官方解法——QUERY 方法**：Body 化的幂等查询。

主持人A：对企业 API 网关、Open API 平台是中长期改造项。**对保险公司的开放平台是值得跟进的标准**——理赔查询、保单详情查询、复杂条件的客户检索，这类"参数复杂 + 幂等"的接口，将来都可以从 POST 改为 QUERY。和 Epic Lore 拼起来——**基础工具栈和底层协议都在被翻新的窗口**。

嘉宾B：好，第三条话题就停在这里。我们切到第四条——今天我个人最想推荐保险公司精算与数据科学团队关注的一个开源项目：**Google Research 的 TimesFM**，时间序列基础模型，今天已经涨到 23087 颗星。

主持人A：先讲一下背景。Foundation Model 这个词大家从 GPT-3 开始就熟悉了——大规模预训练 + 零样本或少样本迁移 = 一个模型解决一类问题。但 Foundation Model 范式过去几年主要在 NLP 领域大放异彩，在时间序列预测领域一直没有真正出圈的开源代表。TimesFM 就是 Google 把"foundation model"范式从 NLP 复制到时间序列预测的一次尝试。

嘉宾B：对保险与金融行业是直接 actionable 的工具。保险公司核心业务里大量是时序问题——**理赔频率预测、车险出险率预测、保费收入预测、欺诈频率监测、退保率预测**——这些都是典型的时序预测。过去十年精算与数据科学团队靠的是 Prophet 加 LSTM 加 ARIMA 这种"碎片化工具栈"，每一个任务都要单独建模、单独调参、单独维护。

主持人A：TimesFM 提供的是"零样本 + 微调"的统一方案。**意味着精算团队可以用一个统一模型解决多个时序预测任务，工程化门槛大幅下降**。我对所有保险公司精算与数据科学部门的建议——**Q3 应该立 PoC**。选一到两个高价值时序预测任务（比如车险出险率预测、欺诈频率监测），用 TimesFM 零样本和微调两种模式，与现有 Prophet / LSTM / ARIMA 模型做对比。**foundation model 范式从 NLP 复制到时序预测的红利期，就在未来 12 个月——早做早受益**。

嘉宾B：好，第四条话题就停在这里。再聊一条今天值得关注的——**Nvidia 让 AI 编程 Agent 自主指挥机器人训练流水线**。这是 Ars Technica 今天报道的新实验，把 AI for Coding 的边界从"写代码"扩张到"控制实验室硬件加设计训练课程加评估机器人技能习得"。

主持人A：这件事我想从"Agent 边界扩张的时间线"来讲。**2024 年是"Agent 写代码"——Cursor、Copilot、Claude Code 起飞**。**2025 年是"Agent 自主开发完整应用"——Devin、Sweep、各种 SWE-Agent 系统**。**2026 年看起来会是"Agent 控制物理实验流水线加操作业务系统 UI"**——Nvidia 让 Agent 控制机器人训练、TesterArmy 让 Agent 自主执行 E2E 测试、UI-TARS-desktop 让 Agent 操作桌面应用、Agent-Reach 让 Agent 抓全网内容。Agent 的能力边界正以季度为单位扩张。

嘉宾B：对保险与风控行业的直接含义有两条——**第一条——机器人保险险种需要重新定义承保范围**。原本"机器人故障导致的财产损失或人身伤害"是一个相对成熟的承保维度。现在加入了一个新维度——**"Agent 自主性导致的故障传播"**。比如一个 AI 编程 Agent 控制机器人训练流水线时出现错误判断、导致连锁故障、波及多台机器人——这种新型故障的因果链很复杂，承保条款需要在未来 24 个月里重新设计。

主持人A：**第二条——内部业务系统的 Agent 自动化将不再局限于"客服加核保"**。可以延伸到"理赔调查加反欺诈线索追踪加监管报告生成"。保险公司里目前 Agent 落地的常见场景是客服意图识别、核保问题答复，相对简单。未来 12 到 18 个月，可以考虑"理赔调查 Agent"——自动拉取理赔单、调取被保险人历史记录、检查异常模式、生成调查建议。这种 Agent 在大型险企里能省下大量的人力成本。

嘉宾B：好，前五个话题我们就聊到这里。接下来进入今天的**企业风控与保险科技专题**——把今天这些信号转化成保险与风控部门可以立即行动的清单。

主持人A：我先开个头。今天我整理了六条 actionable 的建议，每一条都对应今天的真实信号。**第一条——健康险数据流的"Midjourney Medical 适配"准备**。这是今天最重要的信号。健康险与重大疾病险的核保流程，未来 24 个月大概率会出现"医院 → 多模态大模型 API → 结构化报告 → 险企核保系统"的新数据管道。Q3 应该启动技术准备：选一到两家合作医院做 PoC、建立"AI 影像报告"的核保认可机制、和供应商做技术能力评估。**先动手的险企能在 2027 年的健康险定价与核保效率上拿到 12 到 18 个月的领先窗口**。

嘉宾B：**第二条——VMware 替代必须 Q3 启动**。Tesco 把 40000 个负载迁出 VMware 的公开披露是行业最佳实践的拐点信号。保险公司风控、精算、客服三大类数据系统的虚拟化底座，Q3 应启动 PoC 评估——HPE VM Essentials、Proxmox、KVM 是当前三条最现实的替代路径。**错过 2026 年 Q3 到 Q4 的窗口期，2027 年迁移成本会指数级上升**。这件事的紧迫程度，CIO 应该直接列为本季度 Top 3 KPI。

主持人A：**第三条——TimesFM 在精算与风控的 PoC 立项**。Google 开源的 TimesFM 是时间序列基础模型，对保险业的核心预测任务是天然适配的。Q3 精算与数据科学部门应该立 PoC——选一到两个高价值时序预测任务，用 TimesFM 零样本加微调，与现有 Prophet、LSTM、ARIMA 模型做对比。**foundation model 范式从 NLP 复制到时序预测的红利期就在未来 12 个月，早做早受益**。

嘉宾B：**第四条——DeepSeek Vision 替代现有 OCR 工作流的对比实验**。DeepSeek Vision 的成本传统是行业最低之一。Q3 应该立即做对比实验——把现有保单影像 OCR、医疗单据 OCR、ID 影像识别工作流，跑一遍 DeepSeek Vision API 的对比。**预期收益**：成本下降 40% 到 70%、准确率持平或提升、本地化部署能力强（DeepSeek 一贯有本地版本）。**这是边际收益极大、边际成本极小的优化动作**——一两个工程师、两到三周时间就能见到结论。

主持人A：**第五条——把核保和反欺诈 SOP 写成 Claude Skill**。obra/superpowers 与 mattpocock/skills 已经连续两天爆涨——Skills 已经成为新的开源资产形态。把核保规则集、反欺诈话术树、客户回访 SOP、合规检查清单封装为内部 .claude 目录。新员工接入即得能力放大；新险种上线时，把规则更新到 skill 文件比改后台规则引擎更敏捷。**这是 2026 下半年保险公司"内部 AI 工程化"最低门槛的起点**。

嘉宾B：**第六条——后量子密码学迁移的 Q4 规划**。Amazon 加 QuEra 宣称 2028 年实用级量子纠错——比业界共识提前 5 到 7 年。**如果兑现，2028 年应是金融与保险行业完成"后量子密码学迁移"的最后窗口**。法务、安全、IT 三个部门 Q4 应启动联合规划——盘点现有加密体系（保单数据、客户身份、合作方接入接口）、对照 NIST 已批准的 ML-KEM、ML-DSA 等抗量子算法做迁移路径设计。这是"提前 24 个月动手对临时抱佛脚"的典型场景——保单数据是几十年生命周期的资产，加密体系一旦被量子计算攻破就是灾难。

主持人A：把这六条总结一下——**今天这一期，对保险与风控部门来说，是一份"短中长三层并行"的 roadmap**。第一、第四条是季度内可见的快速动作，对应 Midjourney Medical 与 DeepSeek Vision 这两个今日核心信号；第二、第三、第五条是 Q3 启动、Q4 见效的中期改造，对应 VMware 替代、TimesFM 落地、Skill 化 SOP；第六条是三到五年的长期预研，对应后量子密码学。**这种短中长三层并行的 roadmap，是保险公司在 AI 时代真正稳健的发展节奏**。

嘉宾B：而且我想特别强调一点——今天这六条 actionable 建议里，**有五条都不是"追新模型"，而是"用好新工具加扎实做工程"**。这正是昨天我们结尾说的那句话——"AI 标签的红利在退潮，AI 工程化的红利刚刚开始"。**真正决定保险公司未来 12 个月科技竞争力的，不是谁的 AI 故事讲得更响，而是谁把核保数据流、虚拟化底座、时序预测模型、OCR 工作流、内部 SOP、加密体系，这六件"低 AI 含量"的工程改造做得更深**。
