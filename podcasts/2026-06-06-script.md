主持人A：哈喽大家好，欢迎收听每日科技播客，我是晓宇。今天是二零二六年六月六号，星期六。昨天我们聊了 Gemma 4 12B 的开源、Agent 工具链集体上榜、Uber 一千五百美元配额和 Katana BadUSB。今天的故事是接着往下走的——AI 安全从研究室上桌，Agent 中台开始集齐六件套，基础设施开始"就近化"。
嘉宾B：大家好，我是思琪。今天的五条主线我先念一下——第一条是 Anthropic 同一天发两条头条，一条开源"用 AI 找代码漏洞"的参考实现，一条公开 RSI——也就是"AI 改进 AI"——的进度报告，两条并排把今年下半年 AI 安全与 AI 元能力两个最关键的议题摆上桌；第二条是 Agent 中台四件套——MemPalace 的开源记忆系统、CopilotKit 的 AG-UI 协议、Agent-Reach 的跨平台取数 CLI、chopratejas 的 headroom token 压缩，今天把昨天那个"中台拼装"再向前推一格；第三条是 Meta 的 AI 客服 Agent 被攻击者利用导致 Instagram 账号被盗，AI 应用安全第一次有了"业务级事件"案例；第四条是基础设施侧的"就近化"——微软开源 pg_durable 把工作流塞进 Postgres，VoidZero 正式加入 Cloudflare，General Instinct 把 frontier 模型塞进边缘机器人；第五条是应用层在长形态——Vibe-Trading、Open-LLM-VTuber、Mouseless 各自定调。
主持人A：好。先讲今天分量最重的一条，Anthropic 双发。

主持人A：思琪你先讲讲这两条同框的关键。
嘉宾B：好。第一条，[Anthropic 开源的 defending-code-reference-harness](https://github.com/anthropics/defending-code-reference-harness)，HN 头条热度四百九十七，评论一百四十。这个项目的定位是"AI-powered vulnerability discovery framework"——给一份代码仓库，让 Claude 或者其他 LLM Agent 主动找漏洞。这是过去大半年 AI 红队赛道最有分量的一次"基线工程化"——之前大家用 LLM 做白盒安全审查都是各家私有脚本，今天 Anthropic 把官方参考实现开源，企业蓝队第一次有了"用 AI 做白盒"的开源标尺。
主持人A：第二条呢？
嘉宾B：第二条是 [When AI Builds Itself: Our progress toward recursive self-improvement](https://www.anthropic.com/institute/recursive-self-improvement)，Anthropic Institute 把 RSI ——也就是递归自改进——的内部进展公开。热度四百二十七，评论五百六十一——评论数远高于 upvote，说明社区对"AI 自我改进"的安全边界比对技术细节更关心。
主持人A：这两条放在同一天发，是有意还是巧合？
嘉宾B：我倾向认为是有意。Anthropic 一直把 safety 和 capability 当作同一根轴的两端。今天这两条同框发布，相当于在公开声明——"我们既在让 AI 学会找漏洞、保护代码，也在让 AI 学会改进 AI 自身——这两件事必须放在同一个框架下治理"。对企业治理团队来说，这是一个"董事会 AI 委员会议程"必须新增的话题。
主持人A：那企业侧怎么动手？
嘉宾B：三件事。第一，把 defending-code-reference-harness 拉进自己的 CI/CD 流程，作为"AI-Driven SAST"基线——上线前任何对外暴露的 LLM 应用都跑一遍。第二，与现有 Trivy、Snyk、Checkmarx 形成"三层防线"——Trivy 管 SBOM 与 Secret，Snyk 管依赖，defending-code-reference-harness 管"AI 找逻辑漏洞"。第三，治理层面，把"RSI 风险评估"列入年度董事会 AI 报告必备章节——这不是技术问题，这是公司治理问题。
主持人A：这一条已经够沉。咱们换条线。

主持人A：第二条，Agent 中台"四件套"成形。
嘉宾B：今天 GitHub 榜单填齐了我们昨天讲的"Agent 中台"几个关键空位。我先按四件套拆——第一件，记忆层，[MemPalace](https://github.com/MemPalace/mempalace) 五万三千颗星，slogan 直接写"基准最优的开源 AI 记忆系统，免费"。这条与昨天的 Supermemory 形成"双供应商格局"——前者强调闭源生态，后者强调开源 + 基准透明。企业内部 Agent 平台的"记忆层 RFP"今天进入双选阶段。
主持人A：第二件呢？
嘉宾B：第二件，前端协议层，[CopilotKit](https://github.com/CopilotKit/CopilotKit) 三万二千颗星，背后推 AG-UI Protocol——Agent 和 UI 之间的通信协议，React 和 Angular 两条路径都覆盖。这件事意义不输给 MCP 当年——MCP 解决了 "Agent ↔ 工具" 的标准化，AG-UI 想做的是 "Agent ↔ 前端" 的标准化。如果协议立得住，未来三年企业前端栈会出现"AG-UI Native"这个新职业类别。
主持人A：第三件。
嘉宾B：第三件，取数层，[Panniantong 的 Agent-Reach](https://github.com/Panniantong/Agent-Reach) 两万一千颗星——"给 AI Agent 整个互联网的眼睛"。一个 CLI 同时读 / 搜 Twitter、Reddit、YouTube、GitHub、Bilibili、小红书，零 API 费用。这条解决的是过去半年最痛的问题——研报型 Agent 必须跨七八个异构平台取数，每个平台 API 限速不同、收费不同、字段不同，Agent-Reach 把这层做了统一抽象。和昨天的 last30days-skill、Scrapling 一起，今天 Agent 取数链路的"三件套"完整了。
主持人A：第四件。
嘉宾B：第四件，压缩层，[chopratejas 的 headroom](https://github.com/chopratejas/headroom) 一万四千颗星。slogan 一句话——"在工具输出、日志、文件、RAG chunk 进 LLM 之前先压缩"，号称 60-95% 更少 token，答案不变。提供 Library、Proxy、MCP server 三种接入方式。这条直接对应 Agent FinOps——昨天 Uber 一千五百美元配额是"分母收紧"，今天 headroom 是"分子降本"，两端开始合围。一次接入 headroom，你的有效 AI 月度预算可以放大 2 到 10 倍。
主持人A：那加上昨天那几个，整个中台架构是不是六件套了？
嘉宾B：对，正好六件套。Harness（ECC）+ 记忆（MemPalace/Supermemory）+ 前端协议（CopilotKit/AG-UI）+ 取数（Agent-Reach/Scrapling/last30days-skill）+ 压缩（headroom）+ 规范（spec-kit），再加一个形态层（hermes-webui / Open-LLM-VTuber）。这是 2026 年企业自建 Agent 平台的参考架构。各家可以照着抄。

主持人A：第三条最让人警醒——Meta 的 AI 客服 Agent 被攻陷。
嘉宾B：MIT Tech Review 今天的 Download 标题就是 [The Meta hack shows there's more to AI security than Mythos](https://www.technologyreview.com/2026/06/05/1138452/the-download-ai-hacking-mythos-chatbots-brain-impacts/)。报道讲的是——周一有报告披露，攻击者利用 Meta 的 AI 客服 Agent 来盗取 Instagram 账号。攻击路径是这样的：攻击者以"用户找回账号"的名义找 AI 客服，通过精心构造的对话操纵 Agent 调用账号恢复工具，最终把账号过户到攻击者控制下。
主持人A：这是 2026 年第一桩公开的 AI Agent 业务级安全事件吗？
嘉宾B：可以这么说。过去一年我们看到的 AI 安全事件主要分两类——一类是 prompt injection 实验，能让 Agent 说出不该说的话；一类是数据泄露，比如某 SaaS 的训练日志被翻出。但"AI Agent 被攻击者操纵直接造成用户资产损失"，今天是第一桩公开案例。性质完全不同——这意味着 AI 应用安全从理论走入了责任划归。
主持人A：企业侧响应？
嘉宾B：三件事。第一，对外的 LLM Agent 必须做"工具调用授权矩阵"——每一个可调用工具，按用户角色、会话上下文、调用频次三个维度做最小权限。第二，"账户操作类工具"（账号恢复、密码重置、绑定变更、资金操作）必须强制走"人工二次确认"——AI 不能独自完成。第三，"Agent 行为日志"保留时长延长到 180 天以上，且必须达到取证标准——出事后能完整复盘对话上下文和工具调用链。
主持人A：保险侧呢？
嘉宾B：保险侧是个新机会。企业网络安全险里加入"AI Agent 误操作 / 被利用造成用户损失"附加条款；SaaS 厂商责任险增加"AI 客服 Agent 触发的账户安全事件"承保选项；与企业 IT 合同里加入"AI Agent 行为日志保留 ≥ 90 天 + 可审计"硬性要求。今天 Meta 这桩事件，是所有保险公司产品部门今年下半年必学的案例。

主持人A：第四条基础设施。微软 pg_durable、VoidZero 加入 Cloudflare、General Instinct。
嘉宾B：先讲 [pg_durable](https://github.com/microsoft/pg_durable)——HN 热度二百六十二，评论六十九。这是今天分量最被低估的一条。pg_durable 把 Temporal、Cadence、Restate 这一类 "durable workflow execution"——也就是"长时任务、断电可恢复、跨多步异步可重放"的引擎——做成了 Postgres 扩展。
主持人A：意思是工作流跑在数据库里？
嘉宾B：对。过去你要做一个"理赔流程系统"，OCR 一步、外部验真一步、医疗审核一步、再保通知一步——每一步都是异步、可能失败、需要重试、需要事务一致性。传统做法要部署一个 Temporal 集群，里面跑 workflow worker，外面用 Postgres 存业务数据。pg_durable 让你把 workflow 本身也存到 Postgres 里，少一个中间件、少一份运维、少一个 SPOF。对中型理赔系统，是 30% 级的 TCO 改进。
主持人A：然后是 VoidZero 加入 Cloudflare。
嘉宾B：[VoidZero](https://blog.cloudflare.com/voidzero-joins-cloudflare/) 是 Evan You 的公司，Vite 和 Vitest 的母公司，HN 热度六百二十一。VoidZero 加入 Cloudflare 意味着前端工具链和 Edge 平台彻底合拢——Cloudflare Workers 上跑 Vite SSR、跑 Vitest 测试、跑 Agent 推理与 RAG 服务，端到端方案出现。对在 Cloudflare 上构建"全栈 Agent SaaS"的团队，这是个直接利好。
主持人A：General Instinct。
嘉宾B：[General Instinct](https://news.ycombinator.com/item?id=48414869) 是 YC P26 的项目，slogan "Frontier models on edge devices"。两个创始人 Guanming 和 Bill 是机器人背景，他们说——过去几年最佳模型都是按数据中心硬件设计的，但实际机器人硬件是"小算力、小内存、不稳定网络"。General Instinct 在做"把 frontier 级模型塞进边缘机器人"。这个项目接在昨天 Gemma 4 12B "16GB 笔记本可运行" 之后——"frontier 模型 + 边缘硬件"赛道有了 YC 旗手项目。
主持人A：这三条放在一起的解读？
嘉宾B：基础设施"就近化"。pg_durable 把工作流回收到数据库，VoidZero 把前端栈推到 Edge，General Instinct 把模型推到端侧。三个方向同向——AI 时代的中间件正在"中台化、就近化、轻量化"。对运维和 IT 架构师，这是个明确信号——少买一个中间件就是赚到。

主持人A：第五条 Agent 应用层在长形态。Vibe-Trading、Open-LLM-VTuber、Mouseless。
嘉宾B：三个项目各有侧重。[HKUDS/Vibe-Trading](https://github.com/HKUDS/Vibe-Trading) 一万颗星，港大数据科学小组出品，定位是"你的私人 Trading Agent"。这条是资本市场的强信号——Trading Agent 开源进入"主流学术机构挂名"阶段，过去这种项目都是匿名个人作者或不知名小团队，今天是港大背书。
主持人A：那监管侧？
嘉宾B：监管对"AI 自动交易"的责任划归还不完备。可承保侧机会有三块——自营/资管的"AI Trading Agent 故障"责任险、Trading Agent 模型漂移导致回撤超阈值的责任、第三方 Trading Agent SaaS 的供应商责任险。我建议精算和产品部门今年第四季度立项 "Trading Agent 算法责任险" 可行性研究。
主持人A：Open-LLM-VTuber 呢？
嘉宾B：[Open-LLM-VTuber](https://github.com/Open-LLM-VTuber/Open-LLM-VTuber) 九千多颗星——本地、纯离线、带 Live2D 形象、支持语音打断的 LLM 语音交互框架。这条把"本地 Agent + 语音 + 形象"三件事打包，对教育、客服、虚拟陪伴是直接可用的开源底座。商业化的方向是行业模板——比如保险客户经理的"虚拟助理 + 客户应答 + 工单流转"。
主持人A：Mouseless。
嘉宾B：[Mouseless](https://mouseless.click) 是个全平台键盘驱动控制层。表面是个生产力工具，本质是给 Agent 提供"统一的桌面控制 API"——Anthropic computer use、Codex computer use 等场景的开源底盘候选。把它和 AG-UI Protocol 放在一起看，企业内"Agent 直接操作员工桌面"的能力栈正在 ready。

主持人A：好，那咱们进入今天的企业风控与保险科技专题，约五分钟，浓缩成八条可执行建议。
嘉宾B：第一条，AI 应用安全基线——把 Anthropic 的 defending-code-reference-harness 列入必备。所有对外暴露的 LLM 应用，上线前应在它上面跑一遍漏洞挖掘，作为 AI-Driven SAST 基线。和 Trivy 的 SBOM/Secret、Snyk 的依赖扫描形成三层防线。今天 Anthropic 同步发布的 RSI 进展报告，应当作为年度董事会 AI 委员会议程的新增专题——AI 自我改进的风险评估必须制度化。
主持人A：第二条。
嘉宾B：AI Agent 操作责任——把"Agent 误操作"写进保单与合同。Meta 的事件是 2026 年第一桩公开案例，意味着接下来一年类似事件会成串出现。保险公司应主动设计三类新条款——企业网络安全险加入"AI Agent 误操作 / 被利用造成用户损失"附加项；SaaS 厂商责任险增加"AI 客服 Agent 触发账户安全事件"承保选项；与 IT 合同加入"Agent 行为日志保留 ≥ 90 天 + 可审计"硬性要求。
主持人A：第三条。
嘉宾B：Agent 中台采购——六件套统一立项。今天 Agent 中台的参考架构已经成形，保险公司内部 Agent 平台立项时，需要一次性立项 Harness（ECC 思路）、记忆层（MemPalace 与 Supermemory 双供应商）、前端协议（CopilotKit 与 AG-UI）、取数（Agent-Reach + Scrapling）、压缩（headroom）、规范（spec-kit + Spec-Driven）。任一缺一，到生产环境都会变成补丁式架构，三年内必返工。
主持人A：第四条。
嘉宾B：Durable execution 进数据库——理赔系统的减一层架构。[pg_durable](https://github.com/microsoft/pg_durable) 把工作流引擎做成 Postgres 扩展。理赔流程系统在 2026 年第三季度选型时，应作为独立 RFP 项目评估。少一个 SPOF + 少一份运维 + 少一次跨进程事务，对中型理赔系统是 30% 级的 TCO 改进。
主持人A：第五条。
嘉宾B：边缘 frontier 模型——把 General Instinct 拉进 5 年技术雷达。继 Gemma 4 12B 之后，frontier 模型 + 端侧硬件赛道进入投资级阶段。保险公司应在技术雷达上跟踪三类硬件落地——工业机器人、车载推理、家用机器人。这些都是未来五年商业责任险与产品责任险的承保对象，硬件清单越早建越好。
主持人A：第六条。
嘉宾B：Trading Agent 责任险——Vibe-Trading 的监管前瞻。开源 Trading Agent 进入主流学术机构挂名阶段。建议产品和精算部门在第四季度立项"Trading Agent 算法责任险"可行性，覆盖三类风险——自营资管的 Agent 故障、模型漂移导致超阈值回撤、第三方 SaaS 的供应商责任。这是未来三年资本市场风控保险的新品类。
主持人A：第七条。
嘉宾B：基础设施尾部风险——GNSS 干扰与 ISS 漏气。[欧洲 GNSS 干扰源追溯论文](https://arxiv.org/abs/2606.03673) 和 [ISS 空气泄漏避险](https://www.bbc.com/news/live/c4g44ew3g1kt) 同框出现。物流、航运、农机、自动驾驶、商业航天五类业务的尾部风险定价需要刷新——欧洲航线和陆运的 GNSS 失效场景重新建模、商业 LEO 平台的承保条件加入"模块化避险舱"硬性要求、自动驾驶在欧洲市场的"定位回退方案"成为承保条件。
主持人A：第八条。
嘉宾B：AI 健康教练与 AI 客服——交付质量挂钩客户体验险。[Fitbit Air 的 AI 健康教练礼貌过度、价值有限的评测](https://arstechnica.com/gadgets/2026/06/the-fitbit-air-is-great-but-googles-ai-is-too-nice-to-be-your-coach/) 与 [SXSW 上的 AI 依赖度讨论](https://www.technologyreview.com/2026/06/05/1138427/are-ai-chatbots-making-us-lose-control-of-our-brains/) 是同一硬币的两面。健康险与寿险公司在采购 AI 健康教练时，应设立"AI 教练建议错误率"作为 SLA、把"AI 依赖度评估"纳入用户体验研究、合同中保留"AI 模型版本回退 + 人工兜底"条款。
