主持人A：哈喽大家好，欢迎收听每日科技播客，我是晓宇。今天是二零二六年六月二十二号，星期一。我们今天先把今天最有"产业意味"的那一锤丢出来——**字节跳动开源的 deer-flow，一周时间冲到 GitHub 七万两千五百星**。一个月前还没听过这个名字，今天已经稳坐 Trending 第二。更值得注意的是它的定位：**长时程 SuperAgent 框架**，原生集成 sandbox、memory、tools、skill、subagent、message gateway 六件套，目标是承接"分钟级到小时级"的任务。思琪，你怎么看这事？

嘉宾B：哈喽大家好，我是思琪。我看到 deer-flow 这个数字的第一反应是——**中国大厂第一次把"通用 Agent 编排框架"打到 GitHub Trending 前列**。过去这一年，我们聊 Agent 框架，开口闭口都是 LangGraph、CrewAI、AutoGPT、Microsoft AutoGen 这些英语世界的项目。今天 deer-flow 用一个完全独立的技术路径——而且是非常深的技术路径——挤到了世界开源舞台的中央。它跟 LangGraph 那种"状态机编排"和 CrewAI 那种"多智能体协作"不是一个层级的东西，**deer-flow 直接对标的是"完整的 SuperAgent 运行时"**。

主持人A：你说得对。我去翻了一下它的项目结构和文档，最让我意外的是它把六件事放在了一个框架里：**第一是 sandbox**，每个工具调用都跑在隔离环境里；**第二是 memory**，长短期记忆做了分层；**第三是 tools**，传统的函数调用；**第四是 skill**，对接了 Anthropic 那一套 Skill 模式；**第五是 subagent**，可以动态派生子智能体；**第六是 message gateway**，是一个内部的消息总线。**这六件事在以前是六个独立项目，要靠开发者自己拼**。deer-flow 等于把"Agent 时代的 LAMP 栈"打包给你了。

嘉宾B：而且它的"长时程"定位非常关键。我们今天讨论 Agent，绝大多数 demo 都是"几秒到几分钟"的任务——查个天气、搜个资料、写段代码。但企业真正想要 Agent 解决的事，**几乎都是"小时级"的**：写一份尽调报告、跑一遍合同审查、生成一份理赔分析、做一次代码库重构。**这种长时程任务，最难的不是"调通工具"，而是"全程不崩"**——状态恢复、错误回滚、子任务调度、消息持久化，每个环节都可能让 Agent 半路死掉。deer-flow 显然是被这个真实痛点驱动出来的。

主持人A：所以对企业的启示其实很直接——**自研 Agent 平台的选型对照表必须更新了**。原来如果你的对照表只有 LangGraph、AutoGen、CrewAI，那现在 deer-flow 和 obra/superpowers 都要加进去。**而且 deer-flow 是 Apache 协议、字节背书、中文文档优先**——对中国企业来说，落地阻力反而比从英文社区扒一套低得多。我预计下一步会出现一批基于 deer-flow 做行业版的初创——保险尽调版、法律审查版、ESG 报告版。

嘉宾B：完全同意。OK，第二个话题，跟开发者群体相关——**Anthropic 宣布从 7 月 8 日起，使用部分 Claude 能力需要身份证验证**。Hacker News 上这条热度 490，评论 449，几乎所有 AI 开发者都在讨论。这事的意义远超合规本身——**这是 LLM 服务商第一次大规模引入 KYC**。

主持人A：是的，这是个标志性事件。我看到很多开发者第一反应是"这是不是要把个人开发者赶走"，但我觉得**真正的信号其实是另一面**。过去这半年，AI 滥用的舆论压力持续累积——AI 生成的诈骗信息、AI 写的钓鱼邮件、AI 自动化的灰产、各种深度伪造。**KYC 是平台对"AI 滥用→平台担责"压力的第一次系统级回应**。它本质上是在说：从今天起，部分能力你想用，必须有"账号背后是谁"的明确归属。

嘉宾B：而且这事会非常快地推广。**OpenAI、Google 几乎一定会跟进**。一旦行业头部都进入"KYC"模式，企业用 AI 的合规成本曲线会发生有意思的反转——**个人开发者用 LLM 变难，企业用 LLM 反而变容易**。因为企业本来就有完整的 KYC 链路、有合规部门、有法务签字，反而是"实名 + 用途审计"这种链路天然契合的。**这是 AI 治理从"模型对齐"延伸到"账号身份层"的标志**。

主持人A：对企业 IT 部门，我建议立刻做几件事。**第一，把企业内所有 LLM API 账号的归属、用途、调用方梳理一遍**——很多公司的现状是"几个工程师注册了几十个 API Key，散落在各种代码仓库里"，KYC 一启动这种状态全部要重做。**第二，准备一个"多模型 KYC 适配中间件"**——避免被某一家厂商的 KYC 政策推广绑死。**第三，把 LLM 调用纳入 SIEM 日志体系**，便于回查与合规审计。

嘉宾B：你这个"KYC 适配中间件"的提法我觉得是有商业潜力的——**这是一个 12 个月窗口的细分 SaaS 机会**。我们回想 SaaS 爆发期诞生了一批做 SSO、做身份联合的公司，Okta 就是从那个窗口长出来的。**LLM KYC 时代会孕育"AI 时代的 Okta"**——把多家 LLM 厂商的 KYC、配额、计费、审计统一抽象成一个企业网关。

主持人A：好，第三个话题，跟所有做 AI 产品的人都相关——**asgeirtj/system_prompts_leaks 项目，今天单日热度 4.4 万星**。这个项目集中收录了几乎所有主流模型的系统提示词：Anthropic 的 Claude Fable 5、Opus 4.8、Claude Code、Claude Design；OpenAI 的 ChatGPT 5.5 Thinking、GPT 5.5 Instant、Codex；Google 的 Gemini 3.5 Flash、3.1 Pro、Antigravity；xAI 的 Grok；还有 Cursor、Copilot、VS Code、Perplexity 等等。**而且明确写着"定期更新"**。

嘉宾B：这事对企业的冲击有两面。**一面是"教材价值"**——以前做 Prompt 工程，企业团队都是凭直觉、凭经验，现在你可以直接看到顶级团队是怎么写系统提示词的——结构怎么组织、约束怎么表达、安全护栏怎么设、工具调用怎么引导。**这是 Prompt 工程从"民间艺术"走向"教科书化"的一次集中沉淀**。

主持人A：但**另一面是非常严肃的安全教训**——你做的 AI 产品的系统提示词，必须按"已公开"来设计。这件事过去半年我们看过太多教训了：商业秘密被 leak、内部规则被 leak、客户数据样本被 leak。**今天这个项目证明，几乎所有云端 LLM 服务的 system prompt 都可以被逆向并集中开源**。也就是说，**你写在 Prompt 模板里的任何东西，都要假设最终会出现在 GitHub 上**。

嘉宾B：所以对企业 Prompt 安全策略，我给三条非常具体的建议。**第一，企业内部 Prompt 不得包含商业秘密、内部规则、客户数据样本**——这条听起来废话，但真去 audit 一遍你会发现 80% 的企业 Prompt 都违反了。**第二，所有"必须保密"的逻辑放在后端服务里**——Prompt 只能用来"导航"，不能用来"承载机密"。**第三，把"Prompt 安全"纳入数据安全分级**——按"公开数据"管理，这是最稳妥的假设。

主持人A：好，第四个话题，关于安全。今天有两件事我想放在一起讲——**Microsoft 披露的 Crypto Clipper 后门 + iOS 上的 Loupe 应用**。先说 Crypto Clipper：Microsoft 安全团队发现一个新的轻量级后门，**它通过 USB 自我传播，通过 Tor 通信，专门替换剪贴板里的加密货币钱包地址**。这是非常典型的"低调但致命"型攻击——用户复制自己钱包地址转账，结果剪贴板里被悄悄换成了攻击者的地址，钱直接进了别人口袋。

嘉宾B：这事对加密资产托管、交易所、家办、企业财务团队是直接威胁。**而且它揭示了一个被低估的攻击向量——USB 物理介质**。过去十年大家把网络安全的注意力几乎全部放在了"网络入侵"、"钓鱼邮件"、"零日漏洞"上，**但 USB 这个最古老的攻击向量从来没被真正解决过**。尤其在大型企业，开发机、运维机、财务机之间互拷数据的场景仍然非常普遍。

主持人A：对应的安全建议也很清晰。**第一，所有涉及加密资产的内部终端禁用未授权 USB**——这条要写进硬性规范，不是"建议"。**第二，把"剪贴板写入异常"纳入 EDR 规则**——异常的剪贴板替换是有特征的，是可以监控的。**第三，网络安全险条款增加"剪贴板替换型攻击"的明确覆盖语**——这是个细节，但赔付争议往往就在条款细节里。

嘉宾B：对，再说 Loupe。这是一款 iOS 应用，它做的事情非常简单——**让用户"看见"原生 App 在后台到底能看到哪些数据**：通讯录、相册、剪贴板、设备指纹、网络环境等等。Hacker News 热度 506，评论 214。Loupe 之所以引起这么大的讨论，**因为它把一件"大家都知道但没人能直观看到"的事，变成了一个可视化的应用**。

主持人A：对企业的启示有两个层面。**第一是合规层面**——你自己开发的 App，到底在后台默默收集了什么？**很多团队连自己 App 集成的第三方 SDK 在偷偷收集什么都说不清楚**。Loupe 这种工具给企业一个低成本的自查渠道。**第二是商业机会层面**——围绕"第三方 SDK 数据流转治理"，是有清晰付费意愿的 B2B 场景。合规、风控、隐私工程团队都是天然客户。

嘉宾B：好，第五个话题——**Google IPv6 占比突破 50%**，加上 Hacker News 头版一批"AI 时代工程基本功"老文回潮。我把这两件事放在一起，是因为它们指向同一个底层信号——**互联网的"地基"在悄悄重做**。

主持人A：先说 IPv6。Google 的接入流量中 IPv6 占比首次突破 50%，这意味着**未来一两年，IPv4 only 架构在很多场景下会开始出现"用户体验降级"**。对企业的影响是全方位的——CDN 选型、企业出口、WAF 规则、IDS/IPS 规则、日志解析、威胁情报匹配，**所有这些东西的"基线假设"都要更新**。仍然在用 IPv4 only 架构的企业，需要把 IPv6 迁移路线图重新放上桌面。

嘉宾B：再说"AI 时代工程基本功"回潮。今天 Hacker News 头版同时出现了几篇老派工程文章——Sandi Metz 2016 年的《Prefer duplication over the wrong abstraction》、2019 年的《Developers don't understand CORS》、还有《Epoll vs io_uring》。**这不是巧合**。今年 AI 代码助手大规模进入生产，工程师社区开始集体反思一个现象——**AI 生成代码倾向于"过早抽象"、"过度泛化"、对安全机制理解不到位、I/O 模型乱用**。这些老文重提，本质上是社区在主动校准"AI 时代下不能丢的工程价值观"。

主持人A：对企业的启示非常实际——**AI 代码门禁规则要更新**。在 CI 流程里，AI 生成代码应该走"加严"的评审通道，至少检查三件事：**第一，抽象层次是否合理**——别让 AI 把三处类似代码强行抽成一个泛型函数；**第二，安全机制是否到位**——CORS、CSRF、SQL Injection、XSS，AI 经常因为"不知道完整上下文"而漏掉防护；**第三，I/O 模型是否选对**——AI 默认偏好同步模型，但高并发场景下 io_uring 这类模型才是正解。

嘉宾B：而且这事跟我们今天的主题"deer-flow + KYC + Prompt 公开"形成了奇妙的回响——**所有这些事都在告诉企业：AI 时代不是"基础工程不重要了"，反而是"基础工程的价值被重新放大了"**。Agent 框架替你做了编排、KYC 替你做了身份层、Prompt 工程教材替你做了表达技巧，**但是"什么时候不抽象、什么时候选哪种 I/O、什么时候验证 CORS"——这些判断力依然在工程师身上**。

主持人A：说得太好了。OK，那我们进入今天的**企业风控与保险科技专题**——这个板块大约五分钟，我们围绕今天的几件大事聊一聊保险与风控行业的机会、风险与挑战。思琪，先从你最关心的那条开始？

嘉宾B：那我先说**第一个机会——LLM KYC 适配中间件**。Anthropic 这次的 KYC 政策一旦在 OpenAI、Google 跟进，**保险行业其实是最受益的群体之一**。为什么？保险天然是一个高合规要求的行业，从来都是"实名 + 用途审计 + 全链路留痕"。LLM 厂商进入 KYC 时代，保险公司原本就有的合规链路反而可以一比一映射上去。所以保险行业可以做的事是：**把"AI 调用合规中间件"作为下一阶段企业 AI 架构的标配**，包括账号实名映射、调用用途分类、审计日志归档。

主持人A：而且这里有个**保险新险种的机会**——**"AI 服务可用性险"**。一旦 LLM 平台 KYC 推广，某些灰色业务会被批量停摆，但企业的合规业务如果被错杀也会出问题。这中间的损失补偿，可以作为一个新的细分险种，类似云服务可用性险的演化形态。

嘉宾B：**第二个是风险——Crypto Clipper 与剪贴板攻击险**。Microsoft 披露的这个后门特别狡猾，因为它不破坏任何东西、不勒索、不弹窗，**就只是悄悄替换剪贴板里的钱包地址**。这种攻击对企业财务、加密资产托管业务、券商钱包业务的赔付场景是高度具体的。**保险公司应该立刻审视现有网络安全险条款里"剪贴板写入异常引发的资金损失"是否在保障范围**。很多老条款写的是"系统入侵"、"勒索软件"——剪贴板替换严格意义上不属于这两类，可能会出现赔付争议。

主持人A：而且**这个事跟"USB 物理介质"挂钩**——很多企业财务、清算、运维机房还是大量用 USB 互拷文件。保险公司可以推出**"内网物理介质风险险"**，针对那些"无法完全禁用 USB"的高敏感行业（清算所、加密资产托管、能源调度），提供专门的攻击赔付。

嘉宾B：**第三个是挑战——Prompt 安全分级**。system_prompts_leaks 这个事对保险行业的冲击是直接的——**保险公司自研的智能客服、智能核保、智能理赔系统，里面的系统提示词里是不是写了"如果是 VIP 客户就给更宽松的条款"、"如果用户提到某些关键词就触发人工介入"这种规则？这些规则一旦泄露，会直接被竞品和黑产利用**。

主持人A：对，所以保险行业的 Prompt 安全治理建议有两条。**第一，做一次"Prompt 资产盘点"**——把所有线上 AI 产品的系统提示词集中收集、标注敏感级别、设定脱敏改造计划。**第二，把"保密逻辑"从 Prompt 迁移到后端服务**——核保规则、风控逻辑、客户分层这些"必须保密"的东西，应该作为后端服务的判断逻辑，而不是 Prompt 模板里的文字描述。

嘉宾B：**第四个是机遇——长时程 Agent 框架在保险尽调与理赔的应用**。deer-flow 这种"小时级任务"框架，刚好对应保险行业里最难自动化的几个场景——**核保尽调、复杂理赔分析、再保对账**。这些任务的特点是"流程长、状态多、需要回滚、需要审计"，**正是 deer-flow 这种带 sandbox + memory + subagent 的框架最适合的场景**。建议保险公司在 2026 下半年立项做 1-2 个 deer-flow 的内部 PoC，重点跑两个用例：一个是复杂理赔的自动化尽调，一个是再保数据对账。

主持人A：**第五个是趋势——内容真伪鉴定服务**。今天 Hacker News 上还有一条新闻——**一家代理机构整本抄袭畅销作者的书，借 AI 重写后用自己名字重新出版**。这种"AI 加持下的盗版升级"在出版业、内容业、教育业都会快速蔓延。对保险公司而言，**"出版责任险"、"内容真伪鉴定服务"、"AI 抄袭检测保险"会成为未来 12-24 个月的新险种品类**。

嘉宾B：**第六个是基础设施 ready 的风险——IPv6 切换**。Google IPv6 突破 50% 这件事，对保险行业的影响其实非常实际——**核心保单系统、对外 API、客服系统、移动 App 的 IPv6 兼容性，很多老系统其实没测过**。一旦 IPv6 用户体验降级开始集中出现，可能会引发"客户保单查询失败"、"理赔 API 调用失败"这类生产事故。**建议保险公司把"IPv6 ready"作为下一波系统升级的必检项**，纳入年度技术风险评估。
